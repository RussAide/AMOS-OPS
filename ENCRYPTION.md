# Encryption at Rest Strategy — AMOS-OPS

> **Document ID:** D013-04  
> **Status:** Draft — requires implementation before production deployment  
> **Target:** All production environments (on-premise, cloud, hybrid)

---

## 1. Executive Summary

AMOS-OPS handles sensitive Protected Health Information (PHI) and must implement encryption at rest before any production deployment. This document defines the encryption strategy for the SQLite database, backups, and key management.

**Critical requirement:** No production system shall store PHI in an unencrypted database.

---

## 2. Threat Model

| Threat | Risk | Mitigation |
|--------|------|------------|
| Physical device theft | High | Volume-level or database-level encryption |
| Backup media theft | High | Encrypted backups with separate key |
| Unauthorized file access | Medium | File-system permissions + encryption |
| Insider threat (OS-level) | Medium | Encryption keys stored outside the data directory |
| Memory dump analysis | Low | Encrypted pages decrypted only in memory |

## 3. SQLite Encryption Options

### Option A: SQLCipher (Recommended)

[SQLCipher](https://www.zetetic.net/sqlcypher/) is an open-source extension to SQLite that provides transparent 256-bit AES encryption of the entire database file.

```bash
# Installation
npm install better-sqlite3-multiple-ciphers

# Configuration — add to .env
SQLCIPHER_KEY=your-256-bit-key-here-use-crypto-random-bytes
```

**Pros:**
- Transparent to application code — no query changes needed
- 256-bit AES encryption in CBC mode
- Page-level encryption with HMAC verification
- Mature, well-audited codebase

**Cons:**
- Requires rebuilding `better-sqlite3` with SQLCipher support
- Slight performance overhead (~5-15%)

**Migration path:**
1. Export current database to SQL dump
2. Initialize SQLCipher database with key
3. Import dump into encrypted database
4. Update connection string to use SQLCipher
5. Store key in secure secret manager

### Option B: Volume-Level Encryption (Alternative)

Use the operating system's native volume encryption:

| Platform | Technology |
|----------|------------|
| Linux | LUKS / dm-crypt |
| macOS | FileVault (APFS encryption) |
| Windows | BitLocker |
| Docker | Encrypted overlay volumes |

**Pros:**
- No application changes required
- Transparent to SQLite
- Hardware acceleration support

**Cons:**
- Database is decrypted when volume is mounted
- Key management tied to OS/infra
- Less granular than database-level encryption

### Decision: Use BOTH

For defense in depth, the recommended production configuration uses **both** SQLCipher (database-level) and volume-level encryption:

1. **Primary:** SQLCipher encrypts the database file
2. **Secondary:** LUKS/BitLocker encrypts the underlying volume

This ensures that even if one layer is compromised, the data remains protected.

## 4. Key Management Strategy

### 4.1 Key Generation

```bash
# Generate a 256-bit (32-byte) key — run ONCE per environment
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
# Example output: aN9xK3mPqRtUvWxYzAbCdEfGhIjKlMnOpQrStUvWxY=
```

Requirements:
- Minimum 256 bits (32 bytes) of entropy
- Generated by a CSPRNG (`crypto.randomBytes`)
- Never reused across environments (dev, staging, prod each have unique keys)

### 4.2 Key Storage

| Environment | Storage Method | NEVER |
|-------------|---------------|-------|
| Development | `.env` file (gitignored) | Commit `.env` |
| Staging | Docker secrets / environment variables | Hardcode in Dockerfile |
| Production | HashiCorp Vault, AWS KMS, or Azure Key Vault | Store in repository, logs, or config files |

**Production key management workflow:**
1. Key is generated and stored in the secret manager
2. At deployment, the key is injected as an environment variable
3. The application reads the key on startup and keeps it in memory only
4. Key rotation: generate new key, re-encrypt database, update secret manager

### 4.3 Key Rotation

- **Rotation period:** Quarterly (every 90 days) or on suspected compromise
- **Rotation procedure:**
  1. Generate new encryption key
  2. Create fresh encrypted database copy with new key
  3. Verify integrity of re-encrypted copy
  4. Atomically swap to new database file
  5. Update secret manager with new key
  6. Destroy old key after 7-day grace period

## 5. Backup Encryption

### 5.1 Database Backups

All database backups MUST be encrypted before leaving the production server.

```bash
# Encrypted backup script (to be added to cron)
#!/bin/bash
BACKUP_KEY=$(cat /run/secrets/backup-key)
TIMESTAMP=$(date +%Y%m%d_%H%M%S)

# Create encrypted SQL dump
sqlite3 amos-ops.db ".dump" | \
  openssl enc -aes-256-cbc -salt -pass pass:"$BACKUP_KEY" \
  > /backups/amos-ops-${TIMESTAMP}.sql.enc

# Verify backup
openssl enc -aes-256-cbc -d -pass pass:"$BACKUP_KEY" \
  -in /backups/amos-ops-${TIMESTAMP}.sql.enc | \
  sqlite3 /tmp/verify-${TIMESTAMP}.db
```

### 5.2 Backup Key Requirements

- Backup encryption key MUST be different from the database encryption key
- Stored in a separate secret (e.g., `/run/secrets/backup-key`)
- Offsite backups use a different key than local backups

### 5.3 Backup Retention

| Backup Type | Frequency | Retention | Encrypted |
|-------------|-----------|-----------|-----------|
| Full database | Daily | 30 days | Yes (AES-256-CBC) |
| Incremental | Hourly | 7 days | Yes (AES-256-CBC) |
| Offsite archive | Weekly | 1 year | Yes (AES-256-CBC + separate key) |

## 6. Implementation Checklist

Before production deployment, complete all items:

- [ ] Choose SQLCipher or verify volume-level encryption is active
- [ ] Generate production encryption key (256-bit, CSPRNG)
- [ ] Configure secret manager (Vault / AWS KMS / Azure Key Vault)
- [ ] Update database initialization code to support encrypted connections
- [ ] Implement encrypted backup script
- [ ] Generate separate backup encryption key
- [ ] Test backup encryption and decryption cycle
- [ ] Test key rotation procedure on staging environment
- [ ] Document key rotation schedule in operations runbook
- [ ] Train operations staff on key management procedures
- [ ] Add monitoring alert for encryption failures

## 7. Code Changes Required

### Update `api/queries/connection.ts`

Add support for SQLCipher key injection:

```typescript
// In the database connection setup:
const dbKey = process.env.SQLCIPHER_KEY;
if (dbKey) {
  // PRAGMA key must be set immediately after opening the database
  sqlite.exec(`PRAGMA key = "x'${dbKey}"`);

  // Verify encryption is active — this will fail with wrong key
  sqlite.exec("SELECT count(*) FROM sqlite_master");
}
```

### Update `api/lib/env.ts`

Add validation for the encryption key:

```typescript
// Add to env validation:
sqlcipherKey: process.env.SQLCIPHER_KEY,
```

### Update startup sequence (`api/boot.ts`)

Add encryption verification on startup:

```typescript
// After initDatabase():
if (env.isProduction && !process.env.SQLCIPHER_KEY) {
  throw new Error(
    "SECURITY_VIOLATION: SQLCIPHER_KEY is required in production. " +
    "Database encryption at rest is mandatory for PHI. " +
    "See ENCRYPTION.md for setup instructions."
  );
}
```
